                                  # 前言

                                  本文根据 RFC 6749open in new window 撰写，详细介绍了 OAuth2.0 的认证逻辑。

                                  实际使用中根据服务提供者的 API 文档调用对应 API。

                                  # 简介

                                  OAuth2.0 是一个遵循 RFC 6749open in new window 标准的授权协议，用于向第三方应用颁发 Token。

                                  第三方应用可以拿着 Token 去调用对应的 API 获取用户的数据。

                                  # 名词解释

                                  • Client：客户端

                                  • Resource Owner：资源所有者

                                  • Resource Server：资源服务器

                                  • Authorization Server：认证服务器

                                  • Token：令牌

                                    • Access Token：授权令牌
                                    • Refresh Token：刷新令牌

                                  # 角色

                                  在 OAuth2.0 中有四个角色，分别是：客户端 (Client)、资源所有者 (Resource Owner)、资源服务器 (Resource Server)、认证服务器 (Authorization Server) 。

                                  这几个角色的交互流程如下图：

                                  • A => 客户端 (Client) 请求资源所有者 (Resource Owner) 授予权限
                                  • B => 资源所有者 (Resource Owner) 授予客户端 (Client) 权限
                                  • C => 客户端 (Client) 向认证服务器 (Authorization Server) 申请 Token
                                  • D => 经过认证服务器 (Authorization Server) 判断请求为合法后，颁发 Token 给客户端 (Client)
                                  • E => 客户端 (Client) 使用 Token 向资源服务器 (Resource Server) 请求资源
                                  • F => 资源服务器 (Resource Server) 返回资源给客户端 (Client)

                                  其中在 B 中，授权的方式有四种，分别是：

                                  • Authorization Code
                                  • Implicit
                                  • Resource Owner Password Credentials
                                  • Client Credentials

                                  # EndPoints

                                  OAuth2.0 有两个重要的 EndPoint，分别是 Authorization endpoint、Token endpoint，一个可选的 Enpoint，是 Redirection endpoint。

                                  • Authorization endpoint

                                    用于向 Resource Owner 获得授权（需要先通过其他方式认证 Resource Owner 的身份）。

                                    访问这个 Endpoint 时，使用 GET 请求，并在参数中包含 response_type 字段。

                                  • Redirction endpoint

                                    用于认证成功后，将客户端重定向到 Redirect_uri 的地址

                                  • Token endpoint

                                    用于获取 Token 和刷新 Token。

                                    访问这个 Endpoint 时，使用 POST 请求，并在参数中包含 grant_type 字段。

                                  # 授权模式

                                  # 参数说明

                                  • response_type：授权类型

                                    认证页面在发送认证请求的时候，需要加上的参数，用于指定授权模式

                                    • Authorzation Code：code
                                    • Implicit：token

                                  • grant_type：授权类型

                                    客户端在发送获取 Token 请求的时候，需要加上的参数，用于指定授权模式

                                    • Authorzation Code：authorization_code

                                    • Password：password

                                    • Client Credentials：client_credentials

                                  • client_id：客户端 ID

                                    在认证服务器登记时会提供，用于标记客户端。

                                  • redirect_uri：重定向 URL

                                    用于认证服务器返回时，跳转到的 URL

                                    默认会重定向到 OAuth 应用程序设置中配置的回调 URL。 可以在请求时带上一个 URL，这个 URL 的主机和端口必须完全匹配回调 URL。

                                    重定向 URL 的路径必须引用回调 URL 的子目录。

                                    CALLBACK: http://example.com/path

GOOD: http://example.com/path
GOOD: http://example.com/path/subdir/other
BAD:  http://example.com/bar
BAD:  http://example.com/
BAD:  http://example.com:8080/path
BAD:  http://oauth.example.com:8080/path
BAD:  http://example.org

                                  # Authorization Code

                                  Authorization Code 模式是功能最完整、流程最严密的授权模式。

                                  这种模式是最常用的模式，适用于有后端的 Web 服务。

                                  它的特点是用户授权后，会返回一个 Code ，后端通过 Code 获取 Token，这样 Token 就不会经过浏览器，从而保证了 Token 的安全。

                                  在这种模式中，涉及到三个角色进行交互，分别是客户端 (Client) 、资源所有者 (Resource Owner) 、认证服务器 (Authorization Server) 。

                                  • A => 用户访问客户端，选择使用 OAuth 方式登录，客户端会将用户导向认证服务器页面（通常是通过浏览器）

                                    GET /authorize?response_type=code&client_id=s6BhdRkqt3&state=xyz&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com

                                    其中有一些参数：

                                    参数说明是否必填
                                    response_typeAuthorization Code 固定为 “code”是
                                    client_id客户端 ID是
                                    redirect_uri重定向 URL否
                                    scope权限的范围否
                                    state可以填任意值，认证服务器会在返回值附上一模一样的参数值，用于防止跨站攻击否

                                  • B => 用户选择是否授权给客户端

                                    通常需要用户先登录才允许决定是否授权

                                  • C => 认证服务器将用户导向客户端事先指定的重定向 URL (Redirection URL)，同时附上授权码

                                    HTTP/1.1 302 Found
Location: https://client.example.com/cb?code=SplxlOBeZQQYbYS6WxSbIA&state=xyz

                                    其中一些参数：

                                    参数备注是否必填
                                    code授权码是
                                    state如果客户端发送请求时带有这个参数，认证服务器返回一个一模一样的，用于防止跨站攻击否

                                    注意，授权码有效期一般不超过 10 分钟，只允许使用一次，且与 redicrect_uri 和 client_id 是一一对应的关系。

                                  • D => 客户端收到授权码，附上早先的重定向 URL，向认证服务器申请 Token（在后端完成）

                                    POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb
                                    参数备注是否必填
                                    grant_typeAuthorization Code 固定为 “authorization_code”是
                                    code授权码是
                                    redicrect_uri重定向 URL（必须与 A 中填的保持一致）是
                                    client_id客户端 ID是

                                  • E => 认证服务器校验授权码和重定向 URL，确认无误后，向客户端发送 Token (Access Token + Refresh Token)

                                    HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
    "access_token":"2YotnFZFEjr1zCsicMWpAA",
    "token_type":"example",
    "expires_in":3600,
    "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
    "example_parameter":"example_value"
}
                                    参数备注是否必填
                                    access_token授权用的 Token是
                                    token_typeToken 的类型是
                                    expires_inToken 过期时间，如果省略，必须通过其他方式设置过期时间否
                                    refresh_token更新用的 Token否

                                    通过这个 Token，后端可以通过调用对应的 API 获取资源。

                                  # Implicit

                                  Implicit 模式允许直接向前端颁发 Token，不需要经过通过“授权码”这一步。

                                  注意，Implicit 模式没有 Refresh Token。

                                  这种模式适合纯前端应用。

                                  • A => 用户访问客户端，使用 OAuth 方式登录，客户端会将用户导向认证服务器页面（通常是通过浏览器）

                                    GET /authorize?response_type=token&client_id=s6BhdRkqt3&state=xyz&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com
                                    参数备注是否必填
                                    response_typeImplicit 固定为 “token”是
                                    client_id客户端ID是
                                    redirect_uri重定向 URL，需要经过 URL Encode 处理否
                                    scope权限的范围否
                                    state可以填任意值，认证服务器会在返回值附上一模一样的参数值，用于防止跨站攻击否

                                  • B => 用户选择是否授权给客户端

                                    一般需要用户先登录再授权

                                  • C => 认证服务器将用户导向客户端指定的 “重定向 URL ”，并在 URL 的 Hash 部分包含了 Token

                                    HTTP/1.1 302 Found
Location: http://example.com/cb#access_token=2YotnFZFEjr1zCsicMWpAA&state=xyz&token_type=example&expires_in=3600
                                    参数备注是否必填
                                    access_token授权用的 Token是
                                    token_typeToken 的类型是
                                    expires_inToken 过期时间，如果省略，必须通过其他方式设置过期时间否
                                    scope权限范围否
                                    state如果客户端发送请求时带有这个参数，认证服务器返回一个一模一样的，用于防止跨站攻击否

                                    注意，Token 的位置是 URL 锚点（fragment），而不是查询字符串（querystring），这是因为 OAuth 2.0 允许跳转网址是 HTTP 协议，因此存在“中间人攻击”的风险，而浏览器跳转时，锚点不会发到服务器，就减少了泄漏令牌的风险

                                  • D => 浏览器向 Web 资源服务器发出请求，其中不包括上一步收到的Hash值

                                  • E => Web 资源服务器返回一个网页，其中包含的代码可以获取 Hash 值中的 Token

                                  • F => 浏览器执行上一步获得的脚本，提取出 Token

                                  • G => 浏览器将 Token 发给客户端

                                  注意，Web 资源服务器与上面的资源服务器不是同一个服务器。

                                  # Resource Owner Password Credentials

                                  Resource Owner Password Credentials 模式中，用户向客户端提供自己的用户名和密码。客户端使用这些信息向认证服务器获取 Token。

                                  • A => 用户向客户端提供用户名和密码

                                  • B => 客户端将用户名和密码发给认证服务器，向后者请求令牌

                                    这种模式一般直接使用身份验证，比如 HTTP Basic。

                                    POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=password&username=johndoe&password=A3ddj3w
                                    参数备注是否必填
                                    grant_typeResource Owner Password Credentials 固定为 “password”是
                                    username用户名是
                                    password密码是
                                    scope权限范围否

                                  • C => 认证服务器确认无误后，向客户端提供访问令牌

                                    返回的结果如下：

                                    HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
    "access_token":"2YotnFZFEjr1zCsicMWpAA",
    "token_type":"example",
    "expires_in":3600,
    "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
    "example_parameter":"example_value"
}

                                  # Client Credentials

                                  Client Credentials 模式中，客户端以自己的名义向认证服务器认证。

                                  这种适用于没有界面的应用程序。

                                  • A => 客户端向认证服务器进行身份认证，并要求一个访问令牌

                                    发送请求如下：

                                    POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
                                    参数备注是否必填
                                    grant_typeClient Credentials 固定为 “client_credentials”是
                                    scope权限范围否

                                    这种情况下，认证服务器会通过某种方式验证客户端的身份。

                                    在 Postman 中，这种模式使用 HTTP Basic ，即在 Header 中加入 Authorization 字段，用户名和密码填 client_id 和 client_secret。

                                  • B => 认证服务器确认无误后，向客户端提供访问令牌

                                    返回结果如下：

                                    HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
    "access_token":"2YotnFZFEjr1zCsicMWpAA",
    "token_type":"example",
    "expires_in":3600,
    "example_parameter":"example_value"
}

                                  # 使用 Token

                                  Token 的类型一般有两种，一种是 bearer，另一种是 mac

                                  • bearer

                                    GET /resource/1 HTTP/1.1
Host: example.com
Authorization: Bearer mF_9.B5f-4.1JqM

                                  • mac

                                    GET /resource/1 HTTP/1.1
Host: example.com
Authorization: MAC id="h480djs93hd8",
	nonce="274312:dj83hs9s",
	mac="kDZvddkndxvhGRXZhvuDjEWhGeE="

                                  在访问资源服务器提供的 API 时，带上 Token 就可以正常调用。

                                  # 更新 Token

                                  如果用户访问时，客户端的 Access Token 已过期，但 Refresh Token 还没过期，可以通过 Refresh Token 更新 Access Token，而不是重新申请。

                                  POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
                                  参数备注是否必填
                                  grant_type固定为 “refreshtoken”是
                                  refresh_tokenRefresh Token是
                                  scope授权范围，不可以超过上一次的申请的范围，默认与上一次一致是

                                  # 附录

                                  • OAuth2.0 标准

                                    RFC 6749open in new window ：https://tools.ietf.org/html/rfc6749open in new window

                                  • Token (令牌) 与 Password (密码) 的区别

                                    1. Token 可以设置权限的范围，比如只允许写或读，而密码是完整权限的。
                                    2. Token 可以设置有效期，到期自动失效，用户无法自己修改；只要用户不修改密码，密码都是长期有效的。
                                    3. 即使 Token 还在有效期，数据所有者也可以通过撤消使 Token 立即失效，而密码无法撤销，只能更改。

                                    由于 Token 具有访问某些资源的权限，所以 Token 与密码具有同等的重要性。

                                    一般来说，Token 的有效期都设置的非常短，但也要避免 Token 的泄露。
